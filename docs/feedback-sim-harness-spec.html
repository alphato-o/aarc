Harness A — The Whole-Run Feedback Simulator

Fast-forward an entire virtual run through the real voice-generation pipeline — text-only, no ElevenLabs — and read everything Ricky & Jessica would say, before anyone laces up.

Status: Draft v1 — for approval Scope: Harness A only (feedback sim) Sibling: Harness B (iOS end-to-end UI tests) — separate spec Date: 2026-06-16

1Why this exists

Our test loop has been: change code → fly blind → wait for a real run → discover five bugs at once, by ear, mid-suffering → repeat. A real run costs a gym trip and two hours. That makes testing rare, late, and high-stakes — exactly backwards for a system whose whole value is the moment-to-moment quality of two AI voices.

The 12 km treadmill run on 2026-06-16 (24285006) is the case study. Eight defects surfaced at once, and every single voice defect was discoverable offline, deterministically, for free — we just had no way to look:

  • Jessica opened 23 of 23 lines with the identical <hook> — darling, <contempt> template (a template her prompt explicitly bans).
  • Ricky reused one "X minutes in, you've gone QUIET — just the belt…" shape five times.
  • Zero km milestones fired: the script-gen call 502'd three times (script.swapRetry) and the Jessica milestone-delivery path never ran despite director.prewarm km:1…6.
  • The Ricky↔Jessica balance was a flat ~50/50 the whole way — none of the intended "Ricky early → Jessica late" hand-off.
The principle we're adopting: the simulator is the gate; the gym is acceptance, not discovery. We exhaust every offline check before a real run is even on the table. Harness A makes that possible for the voice pipeline.

2Goal & scope

Goal

Given a plan ("10k", "60 min", "open") and a pace profile, drive the real generation pipeline through a fast-forwarded virtual run and emit:

  1. a complete, time-stamped transcript of every line both voices produce, annotated with voice, length mode, trigger/source, the km it belongs to, and the director's decision context; and
  2. an analysis report — repetition score, voice-balance curve, milestone coverage, length distribution, arc check, and any generation failures — rendered as a self-contained HTML page.

In scope

  • The real RunDirector / ScriptEngine / ContextualCoach / Conversation logic.
  • Real LLM generation (so repetition is real), text-only.
  • Deterministic virtual clock + seeded RNG.
  • Headless run on the iOS simulator + an in-app "Preview Run" trigger.

Out of scope (v1)

  • ElevenLabs audio generation — bypassed (text is the artifact). But its latency + playback duration are modelled, because timing shapes the run — see §5.
  • Two fidelity levels: Content mode (v1, idealised timing) and Fidelity mode (v2, full latency/queue simulation). Both specced in §5; v1 ships first.
  • Harness B (iOS UI/e2e) — its own spec.
  • The persona/director fixes themselves — this is the instrument, not the surgery.

3Architecture

The non-negotiable design rule: reuse the exact pipeline the app runs. We do not re-implement the director or the personas in a test double — that's how sim and prod drift. We drive the real singletons through their real entry point (LiveMetricsConsumer.ingest) and intercept only at the very edges: the clock (front) and the speak sink (back).

5Fast-forward, determinism & network variance

How virtual time drives it

The real run is a 1 Hz Timer → ingest. The harness replaces the Timer with a synchronous loop that advances a virtual clock and awaits generation between steps:

// pseudocode — SimRunDriver
RunPreview.active = true
RunDirector.shared.prewarmEnabled = false
start(plan, paceProfile)              // RunOrchestrator/Director/Conversation start as normal
var t = 0.0
while !finished(plan, dist, t) {
    let m = syntheticMetrics(at: t, dist: dist)   // pace injected ⇒ deterministic speed
    LiveMetricsConsumer.shared.ingest(m)          // drives the real pipeline for this tick
    await settleGeneration()                      // await Conversation.produceTask / ScriptEngine in-flight
    t += dt; dist += stepDistance(paceProfile, dt)
}
end()
writeReport(RunPreview.transcript)

Each produced line is tagged with the virtual t at production. Generation is real LLM (~1–3 s/line); a 10 k run yields roughly 40–60 lines, so wall-clock ≈ 1–3 minutes for a full preview — versus a two-hour gym trip. dt can be coarse (e.g. 3–5 s steps) since nothing here needs sub-second fidelity.

Content mode idealises timing — and must not be mistaken for the real timeline. Awaiting the LLM and tagging the line with the virtual t at production collapses latency to ~0. That faithfully reproduces what is said and the director's content decisions, but NOT when it lands, in what order, or whether it's dropped. For that, see "the timing axis" below.

The timing axis — why a single fast-forward isn't enough

A run is a time-ordered event stream, and latency is part of the content. The director pre-warms 25 s ahead, protects 15 s windows, and only lets Jessica react when there's ≥45 s of clear air — all gated on how long generation + TTS actually take. Real numbers from run 24285006 (we log net.req ms per call):

Stagep50p90maxnote
LLM (OpenRouter)4.5 s7.7 s~37 sper line
ElevenLabs TTS11.2 s18.7 s32.3 sscales with length

TTS is length-driven: 14 chars → 1.4 s but ~800 chars → 16–32 s — and the same 828-char line measured 16 s once and 32 s another. So end-to-end lead is routinely 15–50 s, length-dependent, and varies ~2× run-to-run from the network alone. A fixed 6 s lead would never reproduce a guillotined milestone, a late-landing line, or Jessica's indulgent passage eating the next split's window. Timing variance must be modelled, and modelled stochastically.

Two modes

Content mode (v1)Fidelity mode (v2)

Crucially, fidelity mode still uses the real LLM for text, but its real wall-latency is discarded in favour of a sampled virtual latency — the virtual clock advances by the sampled value, the runner keeps moving, and the line becomes "available" at t₀ + sampledLatency. So we see whether the director's projection was right and whether the line still landed in its window, without the sim taking 50 real seconds per line.

The latency model (calibrated from our own logs)

We already emit everything needed — net.req events carry {svc, ms, chars} and (now) the cache verdict. The model samples:

  • LLM latency — distribution by model (p50/p90 from real runs), per call.
  • TTS latency — f(chars) + noise, with the all-important cache-hit ≈ 0 s branch (the R2 cache means a repeated line is instant — the sim must model hit vs miss, since that radically changes timing).
  • TTS audio duration — chars ÷ ~14 chars/s; this plus the queue's 6 s music gap is what gates the next line and decides what goes stale.
  • The hedge — dual-endpoint race + the length-scaled hedge delay (already shipped); modelled so its effect on "time-to-audio" is real.
  • Network profiles — wifi / china-gateway / degraded, each a different distribution, so we can ask "how does the run feel on a bad connection?"

Sampled latencies are fed back into RunDirector.recordPipelineLatency() (:129) so its pipelineLeadSeconds EMA + projected() adapt exactly as on device.

Monte-Carlo — characterise variance, don't fake a single timeline

Because the same input varies ~2×, one timeline lies. Fidelity mode runs K iterations (sampling the latency profile) and reports the distribution of outcomes: milestone-clean rate, guillotine rate, late-land rate, stale-drop rate, and "did the indulgent line steal the next split." A fix is "good" when the bad-outcome rates fall across the profiles, not when one lucky run looks fine.

The enabler (and the main cost of v2): a Clock seam. RunDirector.updateSpeed reads Date() (:215) and VoiceFeedbackQueue uses wall-clock + Task.sleep for the music gap / stale-drop. To run the real queue/director logic on virtual time (rather than re-implement it and reintroduce sim-vs-prod drift), inject a tiny Clock protocol (now() + sleep(_:)) they read: the app supplies the system clock, the sim supplies a virtual one. This is the one real refactor v2 requires — and it's the difference between modelling timing and faking it.

What's deterministic vs not

ComponentDeterministic?Note
Speed / progress / ETA / milestone ownershipYesFrom injected pace + seeded hash. The Date() path in updateSpeed (:215) is bypassed because synthetic metrics always carry currentPaceSecPerKm.
Length-mode pickYesConversation.variation() is a seeded splitmix hash of elapsed/distance — reproducible.

Cost

LLM: yes · TTS: no  ~50 cheap OpenRouter calls per 10 k preview (cents), zero ElevenLabs characters. Optional --mock-llm mode (canned lines) gives a free pass for testing pure pipeline logic — director ownership, length distribution, milestone coverage — without any spend.

HTML report

A self-contained page (same house style as this doc): the verdict banner up top (green/amber/red per metric), the balance + length curves as inline SVG, then the full transcript as a two-column timeline (Ricky left, Jessica right) with km markers and length-mode chips — so the choreography is visible at a glance and the repetition is impossible to miss.

7Invocation

Headless (CI / batch)

Launch on the simulator with SIMCTL_CHILD_AARC_RUN_SIM=10k (+ optional _SEED, _PACE, _MOCK_LLM). The harness runs the preview at launch, writes Documents/run-sim-<plan>.{json,html}, exits. Pull with simctl get_app_container … data — exactly the ShareCardPreviewHarness flow.

In-app (wire to the app)

A "Preview Run" control in the Control Room dev panel: pick plan + pace, tap, watch the transcript stream in, open the report. Same SimRunDriver — this is the "wire it to the actual app so we don't go back and forth" the founder asked for.

9Build plan

Track 1 — Content mode (changes the working rhythm immediately):

  1. P1 — Driver + sink skeleton. SimRunDriver (stepped clock, synthetic metrics), RunPreview recorder, Speaker.speak tap, prewarmEnabled=false, env gate. Output raw JSON transcript. Proves the pipeline runs headless & deterministically.
  2. P2 — Verify against reality. Run the 10k preview; diff its director decisions (ownership curve, ETAs, length modes) against the logged 24285006 events to confirm sim ≡ prod. The trust step.
  3. P3 — Analyzer + HTML report. Repetition score, balance curve, milestone coverage, length dist, failures → the verdict page.

Track 2 — Fidelity mode (models timing & network variance — your call to pull forward):

  1. P4 — Clock seam. Inject the clock protocol into RunDirector + VoiceFeedbackQueue; prod uses the system clock (no behaviour change), the sim a virtual one.
  2. P5 — Latency model + calibration. Build the f(chars) + cache-hit + hedge + network-profile model; calibrate p50/p90 from real net.req logs. Route the sink through the real queue on the virtual clock → played-stream + drops.
  3. P6 — Monte-Carlo + variance report. K-iteration runs; report guillotine / late-land / stale-drop / split-stolen rates per network profile.

Polish: in-app "Preview Run" control; arc metric (once the arc feature lands); --mock-llm free mode.

P1–P3 are the minimum to flip discovery to the machine. P4–P6 are what make the sim trustworthy for timing bugs (drops, guillotines, late lands) — the class you flagged.
